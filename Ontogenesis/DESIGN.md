# Архитектура и дизайн системы Ontogenesis

## 1. Обзор высокоуровневой архитектуры

Ontogenesis построен как модульное приложение с чётким разделением ответственности (Separation of Concerns). Ядро — это **симулятор**, реализующий LCS-моделирование. Вокруг него построены **анализаторы** для выявления паттернов (метаморфозы, фенотипы) и **интерфейсные слои** (API, CLI, будущий GUI).

```
ontogenesis/
├── src/                    # Исходный код на Rust (основной) и Python (анализ)
│   ├── core/              # Ядро симулятора, LCS-логика
│   ├── domains/           # Реализация параметров и правил для 4 доменов
│   ├── phases/            # Логика инициализации/переходов для 5 фаз
│   ├── analysis/          # Алгоритмы анализа (метаморфозы, кластеризация)
│   ├── io/                # Загрузка/сохранение конфигов и данных
│   └── api/               # Внутренние и внешние API
├── data/                  # Статические данные и конфигурации
├── configs/               # Конфигурационные файлы YAML/JSON
├── scripts/               # Вспомогательные скрипты (Python)
├── tests/                 # Модульные и интеграционные тесты
└── docs/                  # Автогенерируемая документация
```

## 2. Детальное описание компонентов

### 2.1. Ядро симулятора (`src/core/`)
*   `simulator.rs`: Главный координационный класс `Simulator`. Управляет жизненным циклом симуляции: инициализация индивида, пошаговое применение LCS-уравнений, обработка событий.
*   `individual.rs`: Структура `Individual`. Содержит векторы значений параметров по доменам за все моменты времени, историю событий и метаданные (ID, пол, фенотип).
*   `lcs_engine.rs`: Чистая математическая реализация Dual LCS. Функции для расчёта `Δy_t` и обновления состояния. Изолирована от доменной логики.
*   `rng_manager.rs`: Централизованный менеджер псевдослучайных чисел (PCG algorithm) для обеспечения воспроизводимости симуляций.

### 2.2. Домены и параметры (`src/domains/`)
Каждый домен — отдельный модуль, реализующий трейт `Domain`.
```rust
pub trait Domain {
    fn get_parameters(&self) -> &HashMap<String, Parameter>;
    fn apply_lcs_step(&mut self, delta_t: f64, rng: &mut RngManager);
    fn handle_cross_domain_influence(&mut self, other_domains: &DomainSet);
    fn is_transition_detected(&self) -> bool;
}
```
*   `morphology.rs`, `physiology.rs`, `psychology.rs`, `sociology.rs`: Конкретные реализации. Содержат логику инициализации параметров, их границы, доменно-специфичные правила (например, пубертатный скачок роста в Morphology).
*   `parameter.rs`: Базовый struct `Parameter`, хранящий текущее и предыдущее значения, ID, калиброванные коэффициенты LCS (`α`, `β`, `γ[]`, `σ_ζ`).

### 2.3. Управление фазами (`src/phases/`)
*   `phase_manager.rs`: Определяет текущую фазу для индивида на основе возраста и (опционально) состояния. Содержит правила для "мягкого" перехода между фазами (плавное изменение некоторых коэффициентов `β` и `γ` за 1-2 года до/после границы).
*   `phase_i.rs`, ..., `phase_v.rs`: Конфигурации, специфичные для фазы: какие параметры активны, какие связи `γ` считаются значимыми, пороги для детекции переходов `κ_d`.

### 2.4. Анализ данных (`src/analysis/` и `scripts/`)
*   `metamorphosis_detector.rs`: Реализует алгоритм из `THEORY.md (Раздел 4)`. Сканирует траекторию индивида, вычисляет `T_d(t)` и идентифицирует окна метаморфозов.
*   `phenotyper.rs`: Классифицирует траекторию старения на Accelerated/Normal/Delayed на основе скорости изменения интегральных маркеров (например, DOSI, FibrosisTracker) в фазах IV-V.
*   `scripts/visualization.py`: Набор Python-скриптов (Matplotlib, Plotly) для построения комплексных графиков: многомерные траектории, heatmap связей, временные разметки метаморфозов.

### 2.5. Ввод-вывод и конфигурация (`src/io/`, `configs/`)
*   `config_loader.rs`: Загружает YAML/JSON конфиги. Используется библиотека `serde`.
*   `individual_serializer.rs`: Сохраняет/загружает состояние `Individual` в компактном бинарном формате (`.bin`) или читаемом JSON.
*   `configs/simulation_default.yaml`: Главный конфигурационный файл. Определяет размер синтетической когорты, длину симуляции, активные домены, флаги (например, `use_generic_social_model`).

### 2.6. API (`src/api/`)
*   `internal_api.rs`: Программный интерфейс для использования симулятора из других Rust-модулей экосистемы LongevityCommon (например, MCOA). Основная функция: `generate_initial_conditions(age: u8) -> HashMap<String, f64>`.
*   `rest_api.rs` (планируется): Thin REST-сервер на `warp` или `axum` для удалённого запуска симуляций и запроса результатов. Пока не реализовано, приоритет низкий.

## 3. Контракты API

### 3.1. Внутренний API для MCOA
```rust
// src/api/internal_api.rs
pub struct OntogenesisAPI;

impl OntogenesisAPI {
    /// Генерирует вектор начальных повреждений для MCOA на заданный возраст.
    /// Выполняет полную симуляцию от рождения до целевого возраста.
    ///
    /// # Аргументы
    /// * `target_age` - Целевой возраст (от 0 до 120) для которого нужны условия.
    /// * `seed` - Опциональное seed для RNG. Если None, используется системное время.
    /// * `config_overrides` - Опциональные переопределения конфига (пол, фенотип и т.д.).
    ///
    /// # Возвращает
    /// `Result<HashMap<String, f64>, SimError>` где ключи — идентификаторы MCOA-счётчиков,
    /// значения — начальные повреждения D_i,0 в диапазоне [0.0, 1.0].
    pub fn generate_mcoa_initial_conditions(
        target_age: u8,
        seed: Option<u64>,
        config_overrides: Option<ConfigPatch>,
    ) -> Result<HashMap<String, f64>, SimError> { ... }
}
```
