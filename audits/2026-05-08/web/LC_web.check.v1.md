## REMAINING_GAPS

1. **Необоснованная смена стека (P0.1)**  
   В исходном проекте используется React + Vite, а `phoenix`-зависимость является клиентской библиотекой для Phoenix Channels (не серверной). План предлагает полную замену на Phoenix LiveView, что не закрывает ни одно из очевидных структурных замечаний, а создаёт высокий риск и требует переучивания команды. Критические замечания (отсутствие API-клиента, тестов, CI, неоптимальный Dockerfile) могут быть устранены без смены стека.

2. **Игнорирование существующей интеграции с Rust-бэкендом**  
   Проект, судя по `RUST_PHOENIX_MIGRATION_PLAN.md`, взаимодействует с Rust-сервером. LiveView потребует переписывания всего взаимодействия с HTTP/WebSocket на серверном Elixir. План не учитывает сложность миграции существующей функциональности страниц.

3. **Отсутствие мер по сохранению текущей функциональности**  
   P0.2 ("перенести существующую функциональность") не детализирован. Нет гарантии, что все React-компоненты (recharts, html-to-image, react-router) будут адекватно заменены на LiveView-эквиваленты без потери возможностей.

4. **Не закрыто критическое замечание — отсутствие API-клиента**  
   P1.2 (создание `api.ex`) правильно адресован, но он привязан к новому стеку. Если стек остаётся React/JS, нужен аналогичный клиент на TypeScript (например, единый модуль Axios с перехватчиками ошибок).

5. **Dockerfile-рефакторинг (P0.3) — хорошо, но неполно**  
   План правильно предлагает вынести nginx-конфиг, но не упоминает multi-stage сборку для React (Node → nginx) — это сохраняется? При переходе на Elixir потребуется другой подход, но если стек остаётся React, текущий Dockerfile уже корректен, и улучшение касается только выноса конфига.

## NOTES

- План P0 (замена на LiveView) является **опциональной архитектурной революцией**, а не ответом на выявленные недостатки аудита. Критические замечания (отсутствие API-клиента, тестов, CI, неоптимальный Dockerfile) решаются без смены стека.
- Если команда действительно хочет мигрировать на Elixir, это должно быть отдельным стратегическим решением, обоснованным в дополнительном документе (ROI, сроки, Skillset). В рамках данного улучшенного плана это **не закрывает критические замечания**, а лишь заменяет их новыми рисками.
- Для утверждения плана необходимо:
  1. Отказаться от P0.1 и P0.2.
  2. Оставить React-стек, реализовать P1 (единый API-клиент на TypeScript, вынос nginx-конфига, engine restrictions — для `package.json` указать `engines`).
  3. Добавить тесты для React-компонентов и API-клиента (аналог P2.3 на Jest/Testing Library).
  4. Реализовать P2.1 (CI) и P2.2 (удаление артефактов).
- План **не закрывает** замечание по отсутствию тестов: P2.3 написан для LiveView, а не для React.
